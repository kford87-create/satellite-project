01 Scope & goals

Replace the spreadsheet-plus-email quoting workflow of small trade shops with a single AI-native prompt-to-quote app. Owner dictates the job, app returns a branded PDF in < 30 seconds, customer accepts by email reply.

In scope for v0

  • Email + password auth, single user per company
  • Setup wizard: rate, markup, tax, logo, service area, trade
  • Prompt-to-Quote screen — NL → structured quote
  • Editable line-item review & branded PDF
  • Email delivery via Resend
  • Quote → Invoice conversion w/ change tracking
  • Manual payment status + nightly overdue cron
  • Per-tenant RAG learning loop
  • Mobile-first responsive PWA

Out of scope (defer)

  • QuickBooks sync, Stripe Connect, online payments
  • SMS, voice input, photo recognition
  • E-signature, customer portal
  • Good / better / best tiering
  • Multi-user, scheduling, dispatch, inventory
  • Native iOS / Android apps
Definition of done. A real contractor can sign up, set up, generate a quote, send it, get it accepted, convert it to an invoice, edit one line, send it, and mark it paid — entirely from a phone — in under 7 minutes. Total infra cost < $50/mo.

03 System architecture

One Next.js app on Vercel, one Supabase project, two external APIs (Claude + OpenAI + Resend). Background work runs on Postgres via graphile-worker; Vercel Cron triggers nightly overdue sweeps.

hot path

Generate quote

PWA → server action → load context → embed → retrieve top-3 → Claude → Zod → insert draft. p95 target < 12s.

async

Embed sent/accepted quote

graphile-worker job fires on status flip, upserts quote_embeddings. Doesn't block UX.

scheduled

Overdue sweep

Vercel Cron @ 02:00 UTC daily flips sent / partial_paid → overdue when due_date < today.

04 Data model

All multi-tenant data scoped by company_id with row-level security. Vector search MUST include the company filter — RLS is the safety net.

Key SQL — vector + RLS

-- Enable pgvector (one-time)
create extension if not exists vector;

-- Per-quote embedding, RLS-scoped to company_id
create table quote_embeddings (
  quote_id            uuid primary key references quotes(id) on delete cascade,
  company_id          uuid not null references companies(id) on delete cascade,
  job_description_emb vector(1536) not null,
  outcome             text not null,
  created_at          timestamptz default now()
);

-- HNSW index for fast cosine similarity
create index on quote_embeddings
  using hnsw (job_description_emb vector_cosine_ops);

-- RLS: queries that forget the filter return zero rows
alter table quote_embeddings enable row level security;
create policy "own embeddings" on quote_embeddings for all
  using (company_id in (select id from companies where user_id = auth.uid()));
Isolation proof. Pre-beta checklist must include running a cross-tenant vector query — a query from company A using company B's typical job description should return zero rows. RLS makes this automatic, but verify it with a real test.

05 Prompt → Quote flow

The hot path. From "Generate Quote" tap to PDF on screen in under 30 seconds, with every quote both learning from and contributing to the contractor's private RAG corpus.

Hot path latency budget

Load settings + context~100 ms
Embed prompt (OpenAI)~250 ms
pgvector top-3 lookup~30 ms
Claude Sonnet generation~6–10 s
Validate + math + insert~80 ms
Total p95 target< 12 s

Failure modes & fallbacks

  • OpenAI down → skip retrieval, use static cold-start few-shot
  • Claude rate-limit → keep typed prompt in localStorage, retry once
  • Zod validation fails → retry once with stricter "JSON ONLY" reminder
  • PDF render fails → save draft, surface "Preview unavailable, retry"
  • Embed job fails → log + retry; never blocks quote send

06 AI learning loop (per-tenant RAG)

Claude is stateless. Each contractor's "personalized AI" is their own historical data, scoped by company_id, injected at generation time. Memory lives in our database.

Two memory stores

structured company_context

Price overrides, default phrases, avg job duration, win-rate buckets, learned warranty text. Updated on every edit.

semantic quote_embeddings

1536-dim embedding of each job_summary + raw_prompt, with the outcome label. Retrieved as few-shot examples.

The learning signal

  • Contractor edits AI-generated quote → price overrides update
  • Contractor marks accepted → outcome flips, gets preferred in retrieval
  • After ~10 quotes, recompute avg job duration per service type
  • After ~10 accepted, warranty text settles on contractor's wording

After ~10 quotes the AI feels eerily like the contractor — because it IS pattern-matching their own past work.

Retrieval at generation time

// /lib/context/load.ts
const { embedding: queryEmb } = await openai.embeddings.create({
  model: 'text-embedding-3-small',
  input: userPrompt,
});

// Top-3 most similar PAST quotes from THIS company only.
// RLS adds the company filter automatically, but we include it explicitly
// as defense-in-depth — never rely on a single layer of isolation.
const fewShot = await db.$queryRaw`
  SELECT q.job_summary, q.line_items, q.total, qe.outcome
  FROM quote_embeddings qe
  JOIN quotes q ON q.id = qe.quote_id
    WHERE qe.company_id = ${companyId}
    AND qe.outcome IN ('sent', 'accepted')
  ORDER BY qe.job_description_emb <=> ${queryEmb}::vector
  LIMIT 3;
`;

The update loop (after every edit)

// /lib/context/update.ts — runs after Save Draft or Send
export async function applyContractorEdits(quoteId: string) {
  const quote = await db.quotes.findUnique({ where: { id: quoteId }});
  const ctx   = await db.company_context.findUnique({
    where: { company_id: quote.company_id },
  });

  // 1. Detect price overrides — any line item whose unit_price differs from
  //    the AI's original suggestion gets stored as a sticky override.
  const overrides = { ...ctx.price_overrides };
  for (const li of quote.line_items) {
    if (li.unit_price !== li.ai_suggested_price) {
      overrides[li.description] = li.unit_price;
    }
  }

  // 2. After 10+ accepted quotes per service type, recompute median duration
  // 3. Capture warranty/payment-terms wording if changed

  await db.company_context.update({
    where: { company_id: quote.company_id },
    data:  { price_overrides: overrides, updated_at: new Date() },
  });
}
Cost reality check. 50 quotes/mo × Claude Sonnet ≈ $2.50. 50 embeddings × OpenAI ≈ $0.001. pgvector storage is free on Supabase. Total AI cost per contractor is ~$2.50/mo against $99–$249 in revenue → 97%+ gross margin on AI.

07 Quote → Invoice flow

Accepted quotes convert to invoices with one click. Line items copy verbatim, contractor edits to reflect unforeseen changes, and the system auto-diffs against the original quote — that delta becomes a transparent "Changes from original quote" section on the invoice PDF.

The delta calculator

// /lib/invoices/changes.ts
type LineItem = { description: string; qty: number; price: number };

export function diffLineItems(
  original: LineItem[],
  current:  LineItem[],
): ChangesFromQuote {
  const byDesc = (arr: LineItem[]) => new Map(arr.map(li => [li.description, li]));
  const A = byDesc(original), B = byDesc(current);

  const added    = current.filter(li => !A.has(li.description));
  const removed  = original.filter(li => !B.has(li.description));
  const modified = current.flatMap(li => {
    const prev = A.get(li.description);
    if (!prev) return [];
    if (prev.qty !== li.qty || prev.price !== li.price) {
      return [{ description: li.description,
                from_qty: prev.qty, to_qty: li.qty,
                from_price: prev.price, to_price: li.price }];
    }
    return [];
  });

  return { added, modified, removed };
}

How the contractor sees it

When the contractor saves an edited invoice, the "Changes from original quote" preview appears live in the UI — exactly what the customer will see on the PDF. No surprise reveals. Transparency is the moat against the contractor-customer trust gap.

9a · Zod schema for AI output

// /lib/claude/schema.ts
import { z } from 'zod';

export const LineItemSchema = z.object({
  category:   z.enum(['equipment', 'labor', 'materials', 'permits', 'other']),
  description:z.string(),
  quantity:   z.number().positive(),
  unit:       z.string(),
  unit_price: z.number().nonnegative(),
  total:      z.number().nonnegative(),
  notes:      z.string().optional(),
});

export const QuoteOutputSchema = z.object({
  job_summary:      z.string(),
  line_items:       z.array(LineItemSchema).min(1),
  warranty_terms:   z.string(),
  payment_terms:    z.string(),
  valid_until_days: z.number().int().positive().default(30),
});

9c · Background embed job (graphile-worker)

// /lib/workers/embed-quote.ts — runs on every status flip to sent/accepted
export async function embedQuote(payload: { quote_id: string }) {
  const q = await db.quotes.findUnique({ where: { id: payload.quote_id }});
  if (!q) return;

  const { data: [{ embedding }] } = await openai.embeddings.create({
    model: 'text-embedding-3-small',
    input: `${q.job_summary}\n\n${q.raw_prompt}`,
  });

  await db.quote_embeddings.upsert({
    where:  { quote_id: q.id },
    create: { quote_id: q.id, company_id: q.company_id,
              job_description_emb: embedding, outcome: q.status },
    update: { job_description_emb: embedding, outcome: q.status },
  });
}

12 Risks & open questions

The places to slow down and re-check before each session, and the questions worth raising with the first 3 customers.

Technical risks

  • Claude latency. Sonnet 4 + 3 few-shots can creep past 12s. Watch prompt size; cache the system prompt.
  • RLS forgetting. A raw SQL query that bypasses the policy is the one bug that ends the company. Always include explicit company_id filter as belt & suspenders.
  • graphile-worker on Vercel. Need to confirm it runs reliably from a Vercel Cron trigger; if not, fall back to a tiny Render or Railway box.
  • React-PDF styling. Less flexible than HTML/Chromium. Have a fallback plan: @vercel/og + Playwright on a separate service if PDFs look amateur.
  • Embedding model lock-in. 1536 dims is OpenAI-specific. Re-embedding the corpus is fine at our scale but worth noting before swapping providers.

Product / GTM open questions

  • Will contractors trust an AI-generated price enough to send it without heavy edits? (First 3 customers answer this.)
  • Is email-reply acceptance ("reply 'I accept'") a real customer signal, or do we need lightweight e-sign sooner?
  • How often do contractors actually edit invoices after sending the quote? (Drives how prominent "Changes from original quote" should be.)
  • QuickBooks copy-paste — is it tolerable, or is QBO sync a v0 must-have to close the first customer?
  • What's the right cap for Solo? 100 quotes/mo might be too high for the lowest-volume customers and too low for the busiest.
Scope discipline. The spec calls out 6 things explicitly out of scope (QBO, SMS, e-sign, Stripe Connect, voice, photos). Each will sound essential when a prospect asks for it. The answer for v0 is "v0.5" — for all of them. Resist scope creep.
