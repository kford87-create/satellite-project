01 Scope & goals

Replace the spreadsheet-plus-email quoting workflow of small trade shops with a single AI-native prompt-to-quote app. Owner dictates the job, app returns a branded PDF in < 30 seconds, customer accepts by email reply.

In scope for v0

  • Email + password auth, multi-user with role-based permissions (1 / 3–4 / 5–10 users per plan tier)
  • Setup wizard: rate, markup, tax, logo, service area, trade
  • SKU catalog ingestion — upload photos / PDFs of supplier price sheets; Claude Vision extracts SKU code, description, unit, price; contractor reviews and approves into catalog
  • Prompt-to-Quote screen — NL → quote that uses the contractor's actual catalog prices, not generic estimates
  • Editable line-item review & branded PDF
  • Email delivery via Resend
  • Quote → Invoice conversion w/ change tracking
  • Manual payment status + nightly overdue cron
  • Per-tenant memory: catalog (primary) + past-quote patterns (secondary)
  • Mobile-first responsive PWA

Out of scope (defer)

  • QuickBooks sync, Stripe Connect, online payments
  • SMS, voice input, photo recognition
  • E-signature, customer portal
  • Good / better / best tiering
  • Scheduling, dispatch, inventory, time tracking
  • Native iOS / Android apps
Definition of done. A real contractor can sign up, set up, generate a quote, send it, get it accepted, convert it to an invoice, edit one line, send it, and mark it paid — entirely from a phone — in under 7 minutes. Total infra cost < $50/mo.
Scope note — multi-user is now in v0. Originally deferred to v0.5, multi-user with role-based permissions has been pulled into v0 to match the pricing tiers (Solo 1 · Crew 3–4 · Shop 5–10). This adds a company_users join table, an invite flow, plan-tier seat caps, and team-scoped RLS — roughly +3 hours of build time and a one-time data-model rewrite of every RLS policy. See §04 and §10 for the implications.

03 System architecture

One Next.js app on Vercel, one Supabase project, three external APIs (Claude + Voyage + Resend — all Anthropic-aligned). Background work runs on Postgres via graphile-worker; Vercel Cron triggers nightly overdue sweeps.

hot path

Generate quote

PWA → server action → load context → embed → retrieve top-3 → Claude → Zod → insert draft. p95 target < 12s.

async

Embed sent/accepted quote

graphile-worker job fires on status flip, upserts quote_embeddings. Doesn't block UX.

scheduled

Overdue sweep

Vercel Cron @ 02:00 UTC daily flips sent / partial_paid → overdue when due_date < today.

onboarding

SKU extraction

Contractor uploads price sheet → Storage → graphile-worker → Claude Vision → review screen → bulk insert into skus → embed each. One-time onboarding cost ~$0.50–$1 per contractor.

retrieval

Per-quote SKU lookup

Every generation: embed prompt → top-10 SKU vector search + top-3 past-quote vector search → both injected. Catalog gives prices; past quotes give style.

trust gate

No fabricated prices

When the AI can't find a matching SKU, it returns the line with unit_price = null and a note to the contractor — never a guessed number.

04 Data model

All multi-tenant data scoped by company_id with row-level security. Vector search MUST include the company filter — RLS is the safety net.

SKU catalog — the contractor's source of truth

-- Each SKU is one line item the contractor can quote.
-- Loaded once via photo / PDF upload, used forever.
create table skus (
  id              uuid primary key default gen_random_uuid(),
  company_id      uuid not null references companies(id) on delete cascade,
  sku_code        text,                              -- "XR16-036" (supplier code, optional)
  description     text not null,                  -- "3-Ton 16 SEER R-410A Condenser, XR16"
  category        text check (category in
                    ('equipment','labor','materials','permits','other')),
  unit            text not null default 'each',
  unit_price      numeric(12,2) not null,
  supplier        text,                              -- "Carrier Distribution"
  notes           text,
  source_doc_id   uuid references sku_documents(id),
  embedding       vector(1024),                      -- for semantic match at quote time
  active          boolean not null default true,
  created_at      timestamptz default now(),
  updated_at      timestamptz default now()
);

-- HNSW index for semantic SKU lookup ("3-ton condenser" → matches XR16-036)
create index on skus using hnsw (embedding vector_cosine_ops);
create index on skus(company_id, active);

-- The uploaded source document (photo or PDF) — kept for audit / re-extraction
create table sku_documents (
  id              uuid primary key default gen_random_uuid(),
  company_id      uuid not null references companies(id) on delete cascade,
  uploaded_by     uuid references auth.users(id),
  file_url        text not null,                   -- Supabase Storage
  file_type       text check (file_type in ('image','pdf')),
  page_count      int,
  status          text check (status in
                    ('processing','review','approved','failed'))
                  default 'processing',
  extraction_log  jsonb,                             -- raw Claude Vision response for audit
  created_at      timestamptz default now()
);

-- RLS — team-scoped (same pattern as everywhere else)
alter table skus enable row level security;
alter table sku_documents enable row level security;
create policy "team skus" on skus for all
  using (company_id in (select company_id from company_users where user_id = auth.uid()));
create policy "team sku_documents" on sku_documents for all
  using (company_id in (select company_id from company_users where user_id = auth.uid()));

-- Enable pgvector (one-time)
create extension if not exists vector;

-- Per-quote embedding, RLS-scoped to the user's company memberships
create table quote_embeddings (
  quote_id            uuid primary key references quotes(id) on delete cascade,
  company_id          uuid not null references companies(id) on delete cascade,
  job_description_emb vector(1024) not null,
  outcome             text not null,
  created_at          timestamptz default now()
);

-- HNSW index for fast cosine similarity
create index on quote_embeddings
  using hnsw (job_description_emb vector_cosine_ops);

-- RLS: every member of a company can read that company's rows
alter table quote_embeddings enable row level security;
create policy "team embeddings" on quote_embeddings for all
  using (company_id in (
    select company_id from company_users where user_id = auth.uid()
  ));

Isolation proof. Pre-beta checklist must include running a cross-tenant vector query — a query from company A using company B's typical job description should return zero rows. RLS makes this automatic, but verify it with a real test.

05 Prompt → Quote flow

The hot path. From "Generate Quote" tap to PDF on screen in under 30 seconds. Every quote pulls exact prices from the contractor's pre-loaded SKU catalog (§5.5) and matches structural patterns from their past accepted quotes.

Hot path latency budget

Load settings + context~100 ms
Embed prompt (Voyage)~250 ms
pgvector SKU top-10 lookup~30 ms
pgvector past-quote top-3 lookup~30 ms
Claude Sonnet generation~6–10 s
Validate + math + insert~80 ms
Total p95 target< 12 s

Failure modes & fallbacks

  • Voyage down → skip retrieval, use static cold-start few-shot
  • Claude rate-limit → keep typed prompt in localStorage, retry once
  • Zod validation fails → retry once with stricter "JSON ONLY" reminder
  • PDF render fails → save draft, surface "Preview unavailable, retry"
  • Embed job fails → log + retry; never blocks quote send

5.5 SKU catalog ingestion — the onboarding-day workflow

Before quoting a single customer, the contractor uploads supplier price sheets (photos or PDFs). Claude Sonnet 4's vision capability extracts every line item — SKU code, description, unit, price — and the contractor approves the extracted catalog in one review pass. From that moment on, every quote pulls exact prices from the contractor's catalog. No generic estimates, no after-the-fact corrections.

The vision extraction call

// /lib/workers/extract-skus.ts — runs in graphile-worker
export async function extractSkus(payload: { doc_id: string }) {
  const doc      = await db.sku_documents.findUnique({ where: { id: payload.doc_id }});
  const fileData = await supabase.storage.from('sku-docs').download(doc.file_url);
  const base64   = Buffer.from(await fileData.arrayBuffer()).toString('base64');

  const mediaType = doc.file_type === 'pdf' ? 'application/pdf' : 'image/jpeg';

  const response = await claude.messages.create({
    model: 'claude-sonnet-4',
    max_tokens: 4096,
    system: [{ type: 'text', text: SKU_EXTRACTION_PROMPT,
               cache_control: { type: 'ephemeral' } }],
    messages: [{
      role: 'user',
      content: [
        { type: 'document', source: { type: 'base64',
                                          media_type: mediaType,
                                          data: base64 } },
        { type: 'text', text: 'Extract every SKU from this price sheet as JSON.' },
      ],
    }],
  });

  const extracted = SkuListSchema.parse(JSON.parse(response.content[0].text));

  // Status flips to 'review' — UI now shows the extracted rows for owner approval
  await db.sku_documents.update({
    where: { id: doc.id },
    data:  { status: 'review', extraction_log: extracted, page_count: response.usage.page_count }
  });
}

The extraction prompt (system block)

export const SKU_EXTRACTION_PROMPT = `
You are extracting SKU line items from a contractor's supplier price sheet
(could be a photo of a printed catalog page or a PDF spec sheet).

For EVERY distinct item you can identify, return one row with:
  - sku_code:    the supplier's part / model number if visible (else null)
  - description: human-readable description as a contractor would say it
                 (include size / capacity / model where shown)
  - category:    one of equipment | labor | materials | permits | other
  - unit:        each | linear ft | sq ft | hour | lb | gallon | lot
  - unit_price:  the price in USD as a number (no $ symbol)
  - supplier:    brand or distributor name if visible
  - notes:       anything important (e.g. "discontinued", "special order")

Rules:
 1. NEVER fabricate prices or SKU codes you cannot read clearly. If unsure,
    set unit_price to null and put your concern in notes.
 2. Skip headers, footers, page numbers, marketing text.
 3. If the same item appears at multiple prices (e.g. tiered), emit one row
    per tier with the tier name in notes.
 4. Normalize units — "ea" → "each", "lf" → "linear ft", "hr" → "hour".
 5. Return ONLY a JSON array. No prose.
`;

Review screen (the trust gate)

OCR + vision extraction is not perfect — the contractor sees every extracted row before it enters the catalog. They can edit any field, mark rows as "skip," or accept the batch as-is. Approved rows insert into skus; rejected rows go to a "needs human" pile (also flips sku_documents.status to approved).

Don't skip the review step. Inserting un-reviewed extracted SKUs is the fastest way to lose contractor trust. A wrong price on a $2,400 condenser is a much bigger problem than typing it in by hand.

Retrieval at quote time

// /lib/context/load.ts — now retrieves SKUs AS WELL AS past-quote patterns
const queryEmb = await embedQuery(userPrompt);

// 1) PRIMARY: matching SKUs from the contractor's catalog (top-10 by similarity)
const matchedSkus = await db.$queryRaw`
  SELECT sku_code, description, category, unit, unit_price, supplier
  FROM skus
  WHERE company_id = ${companyId} AND active = true
  ORDER BY embedding <=> ${queryEmb}::vector
  LIMIT 10;
`;

// 2) SECONDARY: top-3 similar past accepted quotes (for style / bundling patterns)
const fewShot = await db.$queryRaw`
  SELECT q.job_summary, q.line_items
  FROM quote_embeddings qe JOIN quotes q ON q.id = qe.quote_id
  WHERE qe.company_id = ${companyId} AND qe.outcome IN ('sent','accepted')
  ORDER BY qe.job_description_emb <=> ${queryEmb}::vector
  LIMIT 3;
`;

06 Per-tenant memory — catalog + pattern recall

Claude is stateless. Each contractor's "personalized AI" is their own data, scoped by company_id, injected at generation time. Memory lives in our database in two distinct stores with different jobs.

primary

SKU catalog · skus

The contractor's actual price book — ingested up-front from photos / PDFs (see §5.5). Every quote retrieves matching SKUs by vector similarity and Claude composes the quote using these exact prices. This is the source of truth for $$$.

Role: tells the AI what things cost.

secondary

Past-quote patterns · quote_embeddings

1024-dim voyage-3-large embedding of each job_summary + raw_prompt, with outcome. Top-3 similar accepted quotes get retrieved as few-shot examples.

Role: tells the AI how this contractor structures a quote — bundling, line ordering, warranty wording, permit lines, labor estimates.

The clean division of labor. Catalog gives prices. Past quotes give style. Both are scoped to the contractor by RLS. No generic estimates, no "guess and let the user fix it" — the AI only quotes prices that exist in the contractor's catalog.

Falling back when there's no SKU match

If the contractor's prompt mentions something not in their catalog (e.g. "tear out and dispose of old shed"), the AI surfaces it as a line item with unit_price = null and a flag: "No SKU in your catalog for this. Add a price, or we'll skip it." The contractor sets the price in line; saving offers to also persist it as a new SKU. This way the catalog grows organically without ever inserting un-reviewed prices.

Embedding model choice — voyage-3-large over voyage-3

Quotebird optimizes for retrieval quality over cost. With a small per-contractor corpus (10–100 quotes) every retrieved few-shot example has outsized influence on the generated quote — accuracy matters more than saving a few cents.

Sync-to-catalog on line edits (the secondary loop)

// /lib/catalog/sync.ts — invoked when contractor confirms "update catalog?"
export async function syncEditToCatalog(input: {
  company_id: string;
  sku_id?:    string;          // present when line was matched to an existing SKU
  description: string;
  unit:        string;
  unit_price:  number;
  category:    SkuCategory;
}) {
  if (input.sku_id) {
    // existing SKU — update price, re-embed only if description also changed
    await db.skus.update({
      where: { id: input.sku_id },
      data:  { unit_price: input.unit_price, updated_at: new Date() },
    });
  } else {
    // new SKU — insert + enqueue embedding job
    const sku = await db.skus.create({ data: { ...input, active: true } });
    await queue.enqueue('embed-sku', { sku_id: sku.id });
  }
}

Prompt caching — speed lever, not (yet) a cost lever

Anthropic caches unchanging prompt prefixes for 5 minutes (or 1 hour extended TTL) at ~10% the input cost. We turn it on by tagging the static portion of the system prompt with cache_control. End users feel it as faster repeat generations within a session; the business sees a modest cost savings.

07 Quote → Invoice flow

Accepted quotes convert to invoices with one click. Line items copy verbatim, contractor edits to reflect unforeseen changes, and the system auto-diffs against the original quote — that delta becomes a transparent "Changes from original quote" section on the invoice PDF.

The delta calculator

// /lib/invoices/changes.ts
type LineItem = { description: string; qty: number; price: number };

export function diffLineItems(
  original: LineItem[],
  current:  LineItem[],
): ChangesFromQuote {
  const byDesc = (arr: LineItem[]) => new Map(arr.map(li => [li.description, li]));
  const A = byDesc(original), B = byDesc(current);

  const added    = current.filter(li => !A.has(li.description));
  const removed  = original.filter(li => !B.has(li.description));
  const modified = current.flatMap(li => {
    const prev = A.get(li.description);
    if (!prev) return [];
    if (prev.qty !== li.qty || prev.price !== li.price) {
      return [{ description: li.description,
                from_qty: prev.qty, to_qty: li.qty,
                from_price: prev.price, to_price: li.price }];
    }
    return [];
  });

  return { added, modified, removed };
}

How the contractor sees it

When the contractor saves an edited invoice, the "Changes from original quote" preview appears live in the UI — exactly what the customer will see on the PDF. No surprise reveals. Transparency is the moat against the contractor-customer trust gap.

9a · Zod schema for AI output

// /lib/claude/schema.ts
import { z } from 'zod';

export const LineItemSchema = z.object({
  category:   z.enum(['equipment', 'labor', 'materials', 'permits', 'other']),
  description:z.string(),
  quantity:   z.number().positive(),
  unit:       z.string(),
  unit_price: z.number().nonnegative(),
  total:      z.number().nonnegative(),
  notes:      z.string().optional(),
});

export const QuoteOutputSchema = z.object({
  job_summary:      z.string(),
  line_items:       z.array(LineItemSchema).min(1),
  warranty_terms:   z.string(),
  payment_terms:    z.string(),
  valid_until_days: z.number().int().positive().default(30),
});

9c · Background embed job (graphile-worker)

// /lib/workers/embed-quote.ts — runs on every status flip to sent/accepted
export async function embedQuote(payload: { quote_id: string }) {
  const q = await db.quotes.findUnique({ where: { id: payload.quote_id }});
  if (!q) return;

  const { data: [{ embedding }] } = await voyage.embed({
    model: 'voyage-3-large',
    input: [`${q.job_summary}\n\n${q.raw_prompt}`],
    input_type: 'document',
  });

  await db.quote_embeddings.upsert({
    where:  { quote_id: q.id },
    create: { quote_id: q.id, company_id: q.company_id,
              job_description_emb: embedding, outcome: q.status },
    update: { job_description_emb: embedding, outcome: q.status },
  });
}

12 Risks & open questions

The places to slow down and re-check before each session, and the questions worth raising with the first 3 customers.

Technical risks

  • Claude latency. Sonnet 4 + 3 few-shots can creep past 12s. Watch prompt size; cache the system prompt.
  • RLS forgetting. A raw SQL query that bypasses the policy is the one bug that ends the company. Always include explicit company_id filter as belt & suspenders.
  • graphile-worker on Vercel. Need to confirm it runs reliably from a Vercel Cron trigger; if not, fall back to a tiny Render or Railway box.
  • React-PDF styling. Less flexible than HTML/Chromium. Have a fallback plan: @vercel/og + Playwright on a separate service if PDFs look amateur.
  • Embedding model lock-in. 1024 dims is Voyage voyage-3-large-specific. Re-embedding the corpus is fine at our scale (≤100 quotes per contractor) but worth noting before swapping models.

Product / GTM open questions

  • Will contractors trust an AI-generated price enough to send it without heavy edits? (First 3 customers answer this.)
  • Is email-reply acceptance ("reply 'I accept'") a real customer signal, or do we need lightweight e-sign sooner?
  • How often do contractors actually edit invoices after sending the quote? (Drives how prominent "Changes from original quote" should be.)
  • QuickBooks copy-paste — is it tolerable, or is QBO sync a v0 must-have to close the first customer?
  • What's the right cap for Solo? 100 quotes/mo might be too high for the lowest-volume customers and too low for the busiest.
Scope discipline. The spec calls out 6 things explicitly out of scope (QBO, SMS, e-sign, Stripe Connect, voice, photos). Each will sound essential when a prospect asks for it. The answer for v0 is "v0.5" — for all of them. Resist scope creep.
